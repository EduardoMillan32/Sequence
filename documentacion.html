🃏 Sequence Multijugador

Documentación técnica completa del proyecto

JavaScript Vanilla Firebase Realtime DB HTML5 Tiempo Real

Descripción general

Sequence Multijugador es una implementación web del juego de mesa Sequence. Permite que múltiples jugadores se conecten desde distintos dispositivos y jueguen en tiempo real usando Firebase Realtime Database como backend de sincronización.

El proyecto no usa ningún framework de JavaScript. Toda la lógica está escrita en JavaScript Vanilla con variables globales compartidas entre archivos, cargados en orden específico mediante etiquetas <script> en el HTML.

Sin servidor propio: No hay backend personalizado. Firebase actúa como base de datos en tiempo real y todos los clientes (navegadores) reaccionan a los cambios mediante listeners (.on('value'), .on('child_added')). La lógica del juego se ejecuta en el navegador del jugador que sea el anfitrión (el primero en la lista de jugadores).

Arquitectura general

El proyecto tiene 5 archivos JavaScript con responsabilidades bien definidas:

index.html └── carga en orden: config.js → Firebase + reglas + notificaciones lobby.js → Sala de espera + listeners de estado tablero.js → Mapa de cartas + generación del tablero HTML game.js → Lógica del juego (turnos, fichas, sequences) jugador.js → Mano del jugador + modales Dependencias entre archivos: game.js usa → baseDatos (config), miJugador/miJugadorId/jugadoresEnSala (lobby), manoPropia/cartaSeleccionadaIdx/renderizarMano (jugador), mapaCartas/generarTablero (tablero) lobby.js usa → baseDatos/configuracionJuego/inicializarReglas/mostrarToast (config), obtenerMazoBarajado (game) jugador.js usa → miJugadorRef (lobby), configuracionJuego (config)

Estructura de Firebase Realtime Database

Toda la información de la partida vive bajo el nodo sala_activa/:

sala_activa/ ├── jugadores/ │ └── {id_jugador}/ │ ├── nombre: "Carlos" │ ├── color: "azul" │ ├── listo: true │ └── mano: ["AS", "KH", "J1D", ...] │ ├── estado/ │ ├── iniciado: true │ ├── turnoActual: "{id_jugador}" │ ├── ordenTurnos: ["{id1}", "{id2}", ...] │ ├── jugadoresTotales: 2 │ ├── equiposTotales: 2 │ ├── turnosPasados: 0 │ ├── marcaTiempo: 1234567890 │ ├── empate: false │ ├── victoria: "azul" (solo al ganar) │ ├── abandonado: true (solo al abandonar) │ ├── nombreAbandono: "Carlos" │ ├── ultimoJack: { tipo, codigoCarta, jugadorId, ts } │ └── historial/ │ └── {id}: "🃏 Carlos colocó ficha en..." │ ├── tablero/ │ └── {indice}: "azul" (0-99, solo casillas ocupadas) │ ├── mazo/ │ └── [array de cartas restantes] │ └── nombresEquipos/ ├── rojo: "Rojo" ├── azul: "Azul" └── verde: "Verde"

Inicialización de Firebase

El archivo comienza definiendo el objeto firebaseConfig con las credenciales del proyecto en Firebase Console. Estas credenciales son públicas por diseño (Firebase usa reglas de seguridad en el servidor para controlar el acceso, no la ocultación de las credenciales). 

const firebaseConfig = {
  apiKey:            "AIzaSy...",
  authDomain:        "secuence-7d7af.firebaseapp.com",
  databaseURL:       "https://secuence-7d7af-default-rtdb.firebaseio.com/",
  projectId:         "secuence-7d7af",
  storageBucket:     "secuence-7d7af.firebasestorage.app",
  messagingSenderId: "576327423344",
  appId:             "1:576327423344:web:..."
};

firebase.initializeApp(firebaseConfig);
const baseDatos = firebase.database();

baseDatos es la referencia raíz a Firebase Realtime Database. Todos los demás archivos usan esta variable para leer y escribir datos. Es la variable más importante del proyecto.

tablero.js

Mapa de cartas del tablero · Generación del HTML del tablero

Mapa de cartas

mapaCartas es un array de 100 strings que representa el tablero de 10×10 del juego Sequence. Cada posición contiene el código de la carta que ocupa esa casilla (ej: "AS" = As de Espadas, "10H" = 10 de Corazones). Las 4 esquinas son "LIBRE" — casillas especiales que cuentan como ficha para todos los equipos.

Codificación de cartas: El código tiene la forma {valor}{palo}. Palos: S = Spades (♠), H = Hearts (❤️), D = Diamonds (♦️), C = Clubs (♣). Valores: A, 2–10, Q, K. Los Jacks tienen formato especial: J1D (2 ojos), J2H (1 ojo).

Generación visual del tablero

función cartaACodigoAPI(carta)

Convierte el código interno de carta al formato que usa la API de imágenes deckofcardsapi.com. La API usa "0" en lugar de "10" para los dieces. Ejemplo: "10S" → "0S".

función generarTablero()

Limpia el elemento #tablero y recrea las 100 casillas del tablero iterando sobre mapaCartas. Para cada casilla:

  • Crea un <div class="casilla">
  • Asigna data-carta e data-indice como atributos de datos
  • Asigna el onclick que llama a intentarPonerFicha(indice, carta)
  • Si es "LIBRE": agrega clase esquina y muestra ⭐
  • Si es carta normal: crea un <img> con la URL de la API de cartas

Las imágenes usan loading="lazy" para no bloquear la carga inicial y draggable="false" para evitar el comportamiento nativo de arrastre del navegador.

Esta función se llama automáticamente al cargar el archivo y también desde reiniciarEstadoJuegoLocal() al volver al lobby.

Entrada al lobby

función entrarLobby()

Se ejecuta al presionar el botón "Siguiente" en la pantalla de login. Flujo completo:

  1. Valida que el nombre no esté vacío
  2. Busca en localStorage si hay un ID de sesión anterior (sequence_jugador_id). Si existe, elimina ese nodo de Firebase primero (anti-duplicado para recargas de página)
  3. Llama a salaRef.push(miJugador) — crea el nodo del jugador en Firebase en una sola operación atómica, evitando la race condition de push vacío + set separado
  4. Guarda el nuevo ID en localStorage
  5. Registra onDisconnect().remove() para que Firebase elimine el nodo si el jugador se desconecta inesperadamente
  6. Registra onDisconnect().update({ abandonado: true }) en el estado del juego para notificar a los demás
  7. Cambia la pantalla de login a la pantalla de lobby

Selección de color / equipo

función seleccionarColor(color)

Se ejecuta al presionar uno de los botones de equipo (Rojo, Azul, Verde).

  • Actualiza miJugador.color y habilita el botón "Estoy Listo"
  • Reemplaza el título estático "Elige tu equipo" por un <span contenteditable> que permite editar el nombre del equipo en tiempo real
  • El contenteditable dispara cambiarNombreEquipo() en cada tecla, que escribe en sala_activa/nombresEquipos/{color}
  • Marca visualmente el botón del color seleccionado con la clase color-activo
  • Llama a miJugadorRef.set(miJugador) para sincronizar el color con Firebase
función alternarListo()

Alterna el estado listo del jugador entre true y false. Actualiza el texto y color del botón visualmente y sincroniza con Firebase mediante miJugadorRef.set(miJugador).

Listeners de Firebase en el lobby

salaRef.on('value') — Jugadores en sala

Es el listener más importante del lobby. Se dispara cada vez que cualquier dato bajo sala_activa/jugadores/ cambia. Flujo:

  1. Reconstruye jugadoresEnSala filtrando solo nodos con nombre válido
  2. Guard de re-inserción: Si el jugador local (miJugadorId) no aparece en el snapshot (por desconexión temporal de Firebase), lo añade localmente a la lista y lo re-inserta en Firebase con salaRef.child(miJugadorId).set(miJugador)
  3. Limpieza de sala: Si el jugador es el único y aún no limpió (!yaLimpioSala), borra el estado anterior de la partida (tablero, mazo, estado, nombres de equipos)
  4. Llama a actualizarVistaLobby() y verificarReglasParaIniciar()
¿Por qué el guard de re-inserción? Firebase puede ejecutar onDisconnect().remove() si detecta una desconexión temporal de red (por ejemplo, cuando un segundo jugador entra y hay latencia). Sin este guard, el primer jugador desaparecería de la lista hasta que seleccionara color (que llama miJugadorRef.set() y lo restaura). El guard garantiza que el jugador siempre aparezca en su propia lista.

nombresEquiposRef.on('value') — Nombres de equipos

Escucha cambios en sala_activa/nombresEquipos/. Cuando otro jugador edita el nombre de un equipo, este listener actualiza el <span contenteditable> del jugador local, pero solo si el span no está siendo editado activamente (document.activeElement !== spanEditable).

Verificación de reglas para iniciar

función verificarReglasParaIniciar()

Se llama cada vez que el listener de jugadores se dispara. Verifica si se cumplen todas las condiciones para iniciar la partida:

  1. Todos los jugadores deben tener listo: true
  2. Debe haber exactamente 2 o 3 equipos con jugadores
  3. Los equipos deben tener la misma cantidad de jugadores

Si todo es válido, solo el anfitrión (el jugador en posición 0 de la lista) ejecuta la inicialización para evitar que múltiples clientes escriban en Firebase simultáneamente:

  • Construye el orden de turnos intercalando jugadores por equipo (verde → azul → rojo)
  • Elige un jugador aleatorio para el primer turno
  • Llama a obtenerMazoBarajado() y reparte cartas a cada jugador
  • Escribe el mazo restante en sala_activa/mazo/
  • Escribe el estado inicial en sala_activa/estado/ con iniciado: true

Listener de estado del juego

estadoJuegoRef.on('value') escucha sala_activa/estado/ y maneja todos los estados posibles de la partida:

CondiciónAcción
estado.abandonado === trueMuestra pantalla de fin con mensaje de abandono
estado.empate === trueMuestra pantalla de empate técnico
!estado || !estado.iniciadoSi el juego estaba visible, vuelve al lobby y resetea el estado local
estado.victoriaMuestra pantalla de victoria con el nombre del equipo ganador
estado.iniciado === trueTransiciona a la pantalla de juego (con 1 segundo de delay)

Ayudantes de formato visual

función traducirCartaAIcono(codigoCarta)

Convierte un código de carta (ej: "10H") en un fragmento HTML con el número y el emoji del palo, con color según el palo (rojo para corazones/diamantes, gris para espadas/tréboles). Se usa en los mensajes del historial.

función colorearNombre(nombre, color)

Envuelve un nombre en un <b> con el color del equipo y un efecto de brillo (text-shadow). Se usa en todos los mensajes del historial para identificar visualmente a qué equipo pertenece cada acción.

Historial y contador de mazo

baseDatos.ref('sala_activa/estado/historial').on('child_added') escucha cada nuevo mensaje que se agrega al historial. Cuando llega uno, crea un <li> y lo agrega al final de la lista #lista-historial, haciendo scroll automático al fondo.

baseDatos.ref('sala_activa/mazo').on('value') actualiza el contador visual #contador-mazo cada vez que el mazo cambia (cuando alguien roba una carta).

función registrarAccion(mensaje)

Hace push(mensaje) en sala_activa/estado/historial/. Firebase genera un ID único para cada entrada. El listener child_added en todos los clientes recibe el mensaje y lo muestra en pantalla.

Muerte súbita / Empate técnico

El listener de turnosPasados detecta cuando todos los jugadores han pasado su turno consecutivamente (sin hacer ninguna jugada). Cuando pases >= jugadoresEnSala.length, el anfitrión evalúa quién tiene más sequences:

  • Si un equipo tiene más sequences que los demás → escribe victoria: "{equipo}"
  • Si hay empate en sequences → escribe empate: true
  • Si nadie tiene sequences → escribe empate: true
¿Por qué solo el anfitrión? Para evitar que múltiples clientes escriban en Firebase simultáneamente con posibles resultados distintos. El anfitrión es siempre el jugador en posición 0 de jugadoresEnSala, que es el que entró primero a la sala.

Sistema de turnos

El listener de turnoActual se dispara en todos los clientes cada vez que el turno cambia. Compara turnoActualId con miJugadorId para saber si es el turno del jugador local:

  • Es mi turno: Agrega clase mi-turno al panel de mano, actualiza el título con el color del equipo, llama a evaluarOpcionesDeTurno()
  • No es mi turno: Agrega clase esperando-turno, muestra el nombre del jugador que tiene el turno, oculta el botón "Pasar"
función global window.evaluarOpcionesDeTurno()

Analiza la mano del jugador y el estado del tablero para determinar qué opciones tiene:

SituaciónResultado
Tiene al menos una jugada válidaOculta el botón "Pasar"
No tiene jugadas pero tiene cartas muertasOculta "Pasar", muestra toast para descartar
No tiene jugadas ni cartas muertasMuestra el botón "Pasar turno"

Para cada carta en la mano evalúa:

  • J1 (2 ojos): ¿Hay alguna casilla libre en el tablero?
  • J2 (1 ojo): ¿Hay alguna ficha rival no protegida?
  • Carta normal: ¿Hay alguna casilla con esa carta sin ficha? Si todas están ocupadas, es carta muerta.
función pasarTurno()

Calcula el índice del siguiente jugador en listaOrdenTurnos usando módulo (% listaOrdenTurnos.length) para que sea circular. Escribe el ID del siguiente jugador en sala_activa/estado/turnoActual/.

función global window.ejecutarPasoDeTurno()

Se ejecuta cuando el jugador presiona "Pasar ⏭️". Registra la acción en el historial, incrementa turnosPasados usando una transacción Firebase (para evitar condiciones de carrera si dos jugadores pasan al mismo tiempo), y luego llama a pasarTurno().

Colocar y quitar fichas

función intentarPonerFicha(indiceTablero, cartaTablero)

Punto de entrada para todas las jugadas. Se ejecuta al hacer clic en una casilla del tablero. Validaciones en orden:

  1. ¿Es mi turno? Si no → toast de advertencia
  2. ¿Hay carta seleccionada? Si no → toast de advertencia
  3. ¿Es casilla LIBRE (esquina)? → toast informativo
  4. Si la carta es J2 (1 ojo): verifica que haya ficha rival no protegida, muestra overlay, elimina la ficha de Firebase
  5. Si la casilla ya tiene ficha → toast de error
  6. Si la carta no coincide con la casilla (y no es J1) → toast de error
  7. Si todo es válido: coloca la ficha en Firebase, muestra overlay si es J1, registra en historial
función colocarFichaVisual(indice, color)

Se llama desde el listener tableroRef.on('child_added'). Crea el elemento <div class="ficha ficha-{color}"> y lo agrega a la casilla. Usa doble requestAnimationFrame para garantizar que la animación de entrada (entrando → sin clase) funcione correctamente en todos los navegadores. Después de 350ms (cuando la animación de entrada termina), agrega la clase ultima-colocada que produce el brillo, y programa su remoción a los 5 segundos.

función quitarFichaVisual(indice)

Se llama desde el listener tableroRef.on('child_removed'). Anima la ficha con transform: scale(0) y opacity: 0, y al terminar la transición (transitionend) la elimina del DOM y agrega la clase ultima-removida a la casilla (efecto de destello rojo). El efecto dura 5 segundos.

función limpiarEfectosAnteriores()

Cancela los timers activos y quita las clases de efecto visual (ultima-colocada, ultima-removida) de los elementos anteriores. Se llama antes de cada nueva jugada para que solo la última acción tenga el efecto visual activo.

Detección de Sequences

función verificarSequence(colorJugador)

Se llama cada vez que se coloca una ficha. Genera todos los posibles combos de 5 casillas consecutivas en las 4 direcciones y verifica si alguno está completamente ocupado por el equipo:

  • Horizontal: 6 combos por fila × 10 filas = 60 combos
  • Vertical: 6 combos por columna × 10 columnas = 60 combos
  • Diagonal ↘: 6×6 = 36 combos
  • Diagonal ↙: 6×6 = 36 combos

Una casilla "cuenta" para el equipo si tiene una ficha del color del equipo o si es una esquina LIBRE (mapaCartas[indice] === "LIBRE").

función marcarSequence(indices, colorJugador)

Se llama cuando se detecta un combo válido. Proceso:

  1. Genera una clave única ordenando los índices: "5,6,7,8,9"
  2. Si la clave ya está en combosYaMarcados → sale (evita doble conteo)
  3. Cuenta cuántas casillas del combo no estaban ya protegidas. Si son menos de 4 → sale (el combo comparte demasiadas casillas con un sequence anterior)
  4. Agrega la clave a combosYaMarcados
  5. Agrega la clase protegida-{color} a cada casilla del combo (las fichas protegidas no pueden ser removidas por Jacks de 1 ojo)
  6. Incrementa secuenciasLogradas[colorJugador]
  7. Si el equipo alcanzó el número de sequences necesario → escribe victoria: "{color}" en Firebase

Descarte de carta muerta

función global window.intentarDescartarCartaMuerta()

Se ejecuta al presionar "Descartar Carta Muerta 🗑️". Verifica que la carta seleccionada sea efectivamente una carta muerta (todas sus posiciones en el tablero están ocupadas). Los Jacks nunca pueden ser cartas muertas. Si es válida, llama a descartarYRobarSinPasarTurno().

función descartarYRobarSinPasarTurno(cartaDescartada)

Elimina la carta muerta de la mano local, roba una carta del mazo usando una transacción Firebase (para evitar que dos jugadores roben la misma carta), actualiza la mano en Firebase y vuelve a evaluar las opciones de turno. No pasa el turno — el jugador puede seguir jugando con la nueva carta.

función actualizarManoTrasJugada(mensajeHistorial)

Se llama después de cada jugada válida (colocar o quitar ficha). Elimina la carta jugada de la mano local, roba una carta del mazo con transacción Firebase, actualiza la mano en Firebase y llama a pasarTurno().

¿Por qué transacción? Si dos jugadores intentaran robar del mazo al mismo tiempo (imposible en el flujo normal, pero posible por latencia), una transacción garantiza que cada uno obtenga una carta diferente. La transacción es atómica: lee el mazo, extrae la última carta, escribe el mazo reducido — todo en una sola operación.

Reinicio de estado local

función reiniciarEstadoJuegoLocal()

Se llama cuando el juego vuelve al lobby (después de victoria, empate o abandono). Resetea todas las variables locales del juego: contadores de sequences, combos marcados, efectos visuales activos, overlay de Jack, historial visual y regenera el tablero limpio.

función renderizarMano()

Limpia y reconstruye el contenedor #contenedor-cartas con las cartas actuales. Para cada carta:

  • Crea un contenedor .contenedor-carta-individual
  • Crea un <img> con la URL de la API de cartas
  • Si el índice coincide con cartaSeleccionadaIdx, agrega la clase carta-seleccionada
  • Si la carta es J1* (2 ojos), agrega una etiqueta verde "➕ 2 OJOS"
  • Si la carta es J2* (1 ojo), agrega una etiqueta roja "❌ 1 OJO"
  • Asigna onclick que llama a seleccionarCarta(index)
función seleccionarCarta(index)

Si el índice es el mismo que cartaSeleccionadaIdx, lo deselecciona (toggle). Si es diferente, lo selecciona. Luego llama a renderizarMano() para actualizar el estado visual.

Modales de reglas e historial

Los modales se controlan cambiando la clase CSS (oculta-modal) y el display del elemento. Se usan dos mecanismos en paralelo para garantizar compatibilidad con diferentes reglas CSS que puedan estar aplicadas.

FunciónDescripción
abrirReglas()Actualiza el número de sequences necesarios para ganar (#regla-victoria-dinamica) con el valor actual de configuracionJuego.sequencesParaGanar antes de mostrar el modal.
cerrarReglas()Oculta el modal de reglas.
abrirHistorial()Muestra el modal de historial (la lista ya está poblada por el listener de child_added en game.js).
cerrarHistorial()Oculta el modal de historial.
